# 外卖拼单系统 - 测试计划

## 1. 项目概述

本项目为基于 Spring Cloud 微服务架构的外卖拼单系统（Food Pooling System），包含 5 个微服务：

| 服务 | 端口 | 职责 |
|------|------|------|
| food-gateway | 8080 | API 网关，JWT 鉴权，路由转发 |
| user-service | 8001 | 用户注册、登录、地址管理 |
| pool-service | 8002 | 拼单管理、商家/菜品目录 |
| order-service | 8003 | 订单管理 |
| payment-service | 8004 | 支付扣款、退款、结算 |

前端为单页应用（`frontend/index.html`），通过 Python HTTP 服务器（端口 3000）提供静态服务。

系统共暴露 24 个 REST API 端点，其中 7 个为公开接口，17 个需要 JWT 认证。

## 3. 测试策略

### 3.1 测试类型

| 测试类型 | 方法 | 覆盖率目标 |
|----------|------|------------|
| 单元测试 | 针对 Service 层核心逻辑 | 关键路径 100% |
| 集成测试 | API 端到端调用（通过网关） | 24 个端点 100% |
| 确认测试 | 前端 UI 功能走查 | 核心流程 100% |
| 异常测试 | 边界值、异常输入、并发冲突 | 每模块 ≥3 个用例 |

### 3.2 测试方法

- **接口测试**：使用 `curl` 命令行工具通过 Gateway 发送 HTTP 请求

- **集成测试**：依赖 Nacos、MySQL、Redis、RocketMQ 全部启动

## 4. 测试用例设计依据

基于以下设计文档和需求进行用例设计：

1. **API 接口设计**：7 个 Controller 中定义的 24 个 REST 端点
2. **业务流程设计**：
   - 用户流程：注册 → 登录 → 浏览商家 → 选菜 → 创建/加入拼单 → 成团 → 支付
   - 拼单状态机：CREATED → FORMING → FORMED / DISSOLVED
   - 订单状态机：PENDING_CONFIRM → CONFIRMED → CANCELLED
3. **数据模型设计**：Pool、Participant、Item、Order、Payment 等实体关系
4. **前端交互设计**：4 个 Tab（首页/拼单/订单/支付）+ 商家详情 + 购物车

## 6. 通过/失败标准

- **通过**：实际响应与预期一致，状态码正确，数据完整
- **部分通过**：核心功能正确但存在边缘问题
- **失败**：功能不可用或产生错误结果
- **阻塞**：前置条件无法满足，无法执行用例

系统上线标准：所有 P0（阻塞）和 P1（严重）缺陷已修复，P2（一般）缺陷有明确修复计划。
