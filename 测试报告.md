### BUG-001: leavePool 缺少并发保护

**严重程度**：严重

**所在文件**：[PoolServiceImpl.java:192-230](pool-service/src/main/java/com/pool/service/impl/PoolServiceImpl.java#L192-L230)

**缺陷描述**：
`leavePool` 方法使用 `@Transactional` 注解但未使用 Redisson 分布式锁或 CAS 乐观锁。在高并发场景下，多个用户同时退出同一拼单时，`current_members` 和 `current_food_amount` 的减法操作可能出现竞态条件，导致数据不一致。

**问题代码**：
```java
// 没有 lock.tryLock() 保护
poolMapper.update(null, new LambdaUpdateWrapper<PoolEntity>()
        .eq(PoolEntity::getPoolId, poolId)
        .setSql("current_members = current_members - 1")
        .setSql("current_food_amount = current_food_amount - " + foodAmount));
```

**对比**：`joinPool` 方法使用了 Redisson 分布式锁 + CAS 乐观锁双重保护，而 `leavePool` 两者都没有。

**影响**：并发退出时拼单统计（人数、金额）可能不正确。

**修复建议**：添加 Redisson 分布式锁，或使用 CAS 乐观锁（基于 version 字段更新）。

### BUG-002: cancelOrder 无权限校验

**严重程度**：严重

**所在文件**：[OrderServiceImpl.java:190-214](order-service/src/main/java/com/order/service/impl/OrderServiceImpl.java#L190-L214)

**缺陷描述**：
`cancelOrder` 方法接收 `userId` 参数但从未使用它进行权限验证。任何通过 JWT 认证的用户都可以取消系统中任意用户的订单，只要知道 orderId。

**问题代码**：
```java
public void cancelOrder(String orderId, String userId) {
    OrderEntity order = orderMapper.selectOne(...);
    // 缺少：验证 userId 是否是该订单的参与者
    // 缺少：验证 userId 是否有权限取消
    order.setStatus(BusinessConstants.ORDER_STATUS_CANCELLED);
    orderMapper.updateById(order);
}
```

**影响**：严重的越权漏洞，用户可恶意取消他人订单。

**修复建议**：添加权限校验，验证 userId 是否是该订单对应拼单的参与者或发起者。

### BUG-003: refund 接口可重复退款

**严重程度**：严重

**所在文件**：[PaymentServiceImpl.java:92-156](payment-service/src/main/java/com/payment/service/impl/PaymentServiceImpl.java#L92-L156)

**缺陷描述**：
`refund` 方法查询状态为 `SUCCESS` 的 `CHARGE` 记录后进行退款，但退款记录的插入不影响原 CHARGE 记录的状态。因此对同一拼单多次调用 refund 接口，会为同一批扣款记录生成多笔退款，导致资金损失。

**问题流程**：
1. 第一次调用 refund → 查询到 3 条 CHARGE 记录 → 生成 3 条 REFUND 记录 → 退款总额 = 扣款总额
2. 第二次调用 refund → 再次查询到同样的 3 条 CHARGE 记录 → 再次生成 3 条 REFUND 记录 → 重复退款

**影响**：资金安全漏洞，可导致重复退款。

**修复建议**：
1. 退款后将原 CHARGE 记录状态更新为 `REFUNDED`
2. 或添加幂等性检查：先查询是否已存在该拼单的退款批次

### BUG-004: login 无密码/验证码校验

**严重程度**：中等

**所在文件**：[UserServiceImpl.java:47-59](user-service/src/main/java/com/user/service/impl/UserServiceImpl.java#L47-L59)

**缺陷描述**：
`login` 方法仅验证手机号是否存在和用户是否启用，完全不校验密码或短信验证码。前端发送的 `code` 字段在后端被忽略。任何知道他人手机号的用户都可以直接登录他人账户。

**问题代码**：
```java
public UserVO login(LoginRequest request) {
    UserEntity user = userMapper.selectOne(
            new LambdaQueryWrapper<UserEntity>().eq(UserEntity::getPhone, request.getPhone()));
    if (user == null) {
        throw new BusinessException(..., "用户不存在");
    }
    // 缺少：验证码校验
    // 缺少：密码校验
    return buildUserVO(user); // 直接返回 token
}
```

**影响**：认证形同虚设，任何知道手机号的人即可登录。

**修复建议**：至少添加短信验证码校验或密码校验。

### BUG-005: buildPoolDetailVO N+1 查询

**严重程度**：中等

**所在文件**：[PoolServiceImpl.java:287-335](pool-service/src/main/java/com/pool/service/impl/PoolServiceImpl.java#L287-L335)

**缺陷描述**：
`buildPoolDetailVO` 方法先查询参与者列表，然后在循环中为每个参与者单独查询菜品明细。若拼单有 N 个参与者，则产生 1 + N 次数据库查询。

**问题代码**：
```java
List<PoolParticipantEntity> participants = participantMapper.selectList(...); // 查询1
for (PoolParticipantEntity p : participants) {
    List<PoolItemEntity> items = itemMapper.selectList(...); // 查询N
}
```

**影响**：参与人数较多时性能显著下降。

**修复建议**：一次性查询所有菜品（按 poolId），在内存中按 participantId 分组。

### BUG-006: listMyOrders N+1 查询

**严重程度**：中等

**所在文件**：[OrderServiceImpl.java:149-188](order-service/src/main/java/com/order/service/impl/OrderServiceImpl.java#L149-L188)

**缺陷描述**：
`listMyOrders` 方法先查询 settlement 表获取所有用户相关记录，再用 `distinct()` 提取订单 ID，最后查询订单表。当用户参与大量订单时，settlement 表查询可能返回大量数据，且 `.last("LIMIT ...")` 在订单查询上而非 settlement 查询上，可能导致分页不准确。

**影响**：数据量大时性能下降，分页可能不准确。

### BUG-007: 缺少全局异常处理器

**严重程度**：中等

**所在文件**：全局（user-service, pool-service, order-service, payment-service）

**缺陷描述**：
四个服务均未配置 `@RestControllerAdvice` 全局异常处理器。当抛出未预期的异常（如参数类型不匹配、JSON 解析失败）时，Spring Boot 会返回默认的 500 错误页或 Whitelabel Error Page，暴露堆栈信息，且前端无法统一处理。

**影响**：
1. 用户体验差（看到原始错误页面）
2. 可能泄露服务内部信息
3. 前端 `res.code === 0` 判断逻辑在异常时无法正确匹配

**修复建议**：在每个服务或 common 模块中添加 `@RestControllerAdvice` 类。

---

### BUG-008: DTO 缺少参数校验

**严重程度**：低

**所在文件**：所有 Controller 的 @RequestBody 参数

**缺陷描述**：
所有 DTO（RegisterRequest, LoginRequest, CreatePoolRequest, JoinPoolRequest, ChargeRequest 等）均未使用 `@Valid` + `@NotNull`/`@NotBlank`/`@Min` 等 Bean Validation 注解。依赖业务逻辑校验，但业务逻辑可能遗漏某些字段。

**影响**：空值或非法值可能穿透到数据库层。

### BUG-009: addItems 未使用分布式锁

**严重程度**：低

**所在文件**：[PoolServiceImpl.java:453-520](pool-service/src/main/java/com/pool/service/impl/PoolServiceImpl.java#L453-L520)

**缺陷描述**：
`addItems` 方法使用了 CAS 乐观锁更新池统计，但没有像 `joinPool` 那样使用 Redisson 分布式锁保护整个操作流程。在极端并发下，两个用户同时调用 `addItems`，虽然 CAS 保护了 `current_food_amount` 更新，但菜品记录已先于 CAS 检查插入，若 CAS 失败，菜品记录不会回滚（事务回滚时会，但在 retry 场景下逻辑不一致）。

**影响**：边缘场景下的数据一致性风险。

### BUG-011: 前端页面返回栈异常

**严重程度**：低

**所在文件**：[index.html:451-464](frontend/index.html#L451-L464)

**缺陷描述**：
`doCreatePool` 中设置 `pageStack = ['main']` 以在创建后跳转详情页时返回到主页，但如果用户从商家详情页进入创建页，商家详情页已被压入栈中，`pageStack = ['main']` 会导致之前的导航历史丢失。另外 `showPoolDetail` 内调用 `navigate('pool-detail')` 但 `doJoinPool` 中使用 `pageStack.pop()` 和 `showPoolDetail`，跳转逻辑不一致。

**影响**：用户在特定操作路径下按返回按钮可能跳转到非预期页面。

### BUG-012: inviteCode 使用 Random 生成

**严重程度**：低

**所在文件**：[PoolServiceImpl.java:522-530](pool-service/src/main/java/com/pool/service/impl/PoolServiceImpl.java#L522-L530)

**缺陷描述**：
邀请码使用 `java.util.Random` 生成，在同一毫秒内创建多个拼单时可能生成相同邀请码。虽然概率低，但未做唯一性检查。

**修复建议**：使用 `SecureRandom` 或在生成后检查数据库唯一性。

## 7. 测试结论

### 7.1 系统优点

1. **微服务架构清晰**：5 个服务职责分明，通过 Gateway 统一入口
2. **JWT 鉴权完善**：网关层统一鉴权，白名单配置合理，73 个鉴权相关测试用例通过率 100%
3. **核心拼单流程可用**：创建→加入→成团→下单→扣款的主流程正常工作
4. **并发保护有考虑**：joinPool 使用 Redisson + CAS 双重保护
5. **前端 UI 美观**：美团风格界面，分类浏览、购物车、商家详情等体验流畅

### 7.2 主要问题

1. **安全漏洞**：3 个严重缺陷涉及权限和资金安全（BUG-001, BUG-002, BUG-003）
2. **认证缺失**：login 接口无密码/验证码校验（BUG-004）
3. **并发保护不一致**：joinPool 有锁但 leavePool 无锁（BUG-001）
4. **接口幂等性不足**：退款接口可重复执行（BUG-003）
5. **异常处理缺失**：无全局异常处理器（BUG-007）

### 7.3 上线建议

| 优先级 | 问题 | 必须修复才能上线 |
|--------|------|------------------|
| P0 | BUG-002 cancelOrder 越权 | 是 |
| P0 | BUG-003 重复退款 | 是 |
| P0 | BUG-004 无认证登录 | 是 |
| P1 | BUG-001 leavePool 并发安全 | 是 |
| P1 | BUG-007 全局异常处理 | 建议修复 |
| P2 | BUG-005/006 N+1 查询 | 可后续优化 |
| P2 | BUG-008 参数校验 | 可后续修复 |
| P3 | BUG-009~012 | 可后续修复 |

**结论**：系统核心功能基本可用，但存在 3 个严重安全缺陷（权限越权、重复退款、无认证登录）必须在上线前修复。建议优先处理 P0 级缺陷后进行回归测试。
